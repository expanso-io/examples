# telemetry-cost

Run the telemetry cost arguments from the blog series yourself.

The series claims that most observability spend is a data movement tax: you
pay to ship, index, and store log lines that nobody will ever query. This
example lets you check that claim instead of trusting it. Real log formats
stream in from the public
[log-simulators](https://github.com/expanso-io/log-simulators) suite, Expanso
Edge pipelines parse, filter, and route them at the edge, and a live dashboard
meters every byte into a running dollar figure. Four demos, one per post: the
tax, the audit, the filter, the tiers.

You run it through [Expanso Cloud](https://cloud.expanso.io), the same motion
you would use on a real fleet. You deploy a job once to the cloud control
plane; the cloud schedules it onto your edge nodes and manages its lifecycle.
Here the node is your laptop, a real registered node that just happens to be
nearby, so the log streams and the dashboard are local while the orchestration
is genuinely remote. Every pipeline is a standard Expanso job YAML, and the same
file that lands on your laptop lands on a fleet of 400 the same way.

New here? **[QUICKSTART.md](QUICKSTART.md) is the three-step front door.**

## The event source: real formats, not toy JSON

The demos do not invent a convenient schema. They ingest four streams in the
formats your fleet actually emits, generated by
[expanso-io/log-simulators](https://github.com/expanso-io/log-simulators)
(one `uvx` command each, no install):

| Stream | Command | Format |
|---|---|---|
| App logs | `uvx --from git+https://github.com/expanso-io/log-simulators logsim-app` | Structured JSON lines with trace IDs, latencies, errors |
| Kubernetes | `uvx --from git+https://github.com/expanso-io/log-simulators logsim-k8s --scenario crash-loop` | CRI container lines (zap JSON, klog, ingress access) with a crash-looping pod |
| Web servers | `uvx --from git+https://github.com/expanso-io/log-simulators logsim-web` | NCSA combined access logs |
| Cloud audit | `uvx --from git+https://github.com/expanso-io/log-simulators logsim-cloud` | AWS CloudTrail JSON events |

The pipelines detect and parse each format at the edge: NCSA regex for web,
CRI prefix plus embedded zap/klog for Kubernetes, CloudTrail and app JSON by
shape. The classification rules live in one shared Bloblang block in the job
YAMLs, with a reference implementation in `scripts/classify.py`.

`make sims` starts all four streams locally; `make fixtures` regenerates the
committed, seeded fixture files (byte-identical on every machine).

## Architecture

Two planes. The **control plane** (Expanso Cloud) holds your jobs and decides
which nodes run them. The **data plane** (your node, the simulators, and the
costboard) runs the pipeline next to the data. You deploy once to the cloud; the
cloud places the work on the node. You never SSH the node.

```
            CONTROL PLANE  ............  cloud.expanso.io
            you deploy a job once; the cloud schedules it
            onto nodes matching demo=telemetry-cost
                          |
                          |  job placement (remote)
                          v
========================= DATA PLANE (your node) ===================

logsim-app   logsim-k8s   logsim-web   logsim-cloud
     \            |            |            /
      \           |  TCP       |           /
       v          v            v          v
+---------------------------------------------------+
|  Expanso Edge                                     |
|                                                   |
|  jobs/00-intake.yaml    socket_server :5601       |
|    pure tee, no parsing:                          |
|      raw copy --------------> costboard /ingest/raw
|      raw copy -----+                              |
|                    v                              |
|  jobs/01..04       http_server :8081              |
|    parse NCSA / CRI / CloudTrail / JSON,          |
|    classify, filter, dedupe, route;               |
|    then ship the ORIGINAL RAW LINE                |
+--------+--------------------------+---------------+
         | hot lane                 | cold lane
         v                          v
+---------------------------------------------------+
|  costboard :8090                                  |
|  /ingest/raw     /ingest/hot      /ingest/cold    |
|  dollar meter, per-lane volume, reduction %       |
+---------------------------------------------------+
                                    |
                                    v
                  cold-storage/YYYY/MM/DD/HH/*.jsonl.gz
                  (rehydrate with scripts/rehydrate.py)
```

The intake job tees every raw line twice: one copy to the costboard (so you
always see what an unfiltered architecture would have billed) and one copy
into the scenario pipeline on port 8081. The scenarios parse and classify to
make decisions, but whatever survives ships as the original raw line, so the
billing math is byte-honest. The gap between raw and hot is the money.

## Quickstart (Expanso Cloud)

The headline path. You register your laptop as one edge node, then deploy the
pipelines to it through the cloud. Full walkthrough in
[QUICKSTART.md](QUICKSTART.md).

```bash
# install uv, expanso-edge, expanso-cli (see QUICKSTART.md Step 0)
git clone https://github.com/expanso-io/examples.git
cd examples/telemetry-cost

make cloud-setup    # one-time: create a network at cloud.expanso.io,
                    # bootstrap this laptop as a demo node, save a CLI profile
make demo           # the cloud schedules the jobs onto your node;
                    # watch the dollar meter at http://localhost:8090
```

`make demo SCENARIO=tax` (or `audit`, `filter`, `tiers`) runs one scenario.
`make demo SCENARIO=filter NOPAUSE=1` is the no-pause B-roll mode for clean
screen recordings. `make doctor` checks tools and cloud connectivity if a run
will not start. `make clean` stops everything and leaves your node and profile
in place (reused next run).

The demo node is dedicated: it carries the label `demo=telemetry-cost` and
lives in its own data dir, and the jobs select on that label, so the work never
lands on any other Expanso node you may have on this machine or network.

## Quickstart (local, offline)

No account needed. A local Expanso Edge node in `--local` mode plays the fleet;
this is also what CI and the eval harness use.

```bash
# install uv, expanso-edge, expanso-cli (see QUICKSTART.md Step 0)
git clone https://github.com/expanso-io/examples.git
cd examples/telemetry-cost

make demo-local     # local edge + dashboard, same scenarios, same numbers
```

Requires Linux or macOS, Python 3.10+, `make`, and
[uv](https://docs.astral.sh/uv/) (for `uvx`). Same dashboard at
http://localhost:8090.

## Quickstart (Docker, local)

A containerized version of the offline path. The simulators, the edge node, and
the dashboard all run as containers; the orchestration is local to compose.

```bash
git clone https://github.com/expanso-io/examples.git
cd examples/telemetry-cost
docker compose --profile demo1 up
```

Swap the profile for `demo2`, `demo3`, or `demo4`. The costboard is on
http://localhost:8090 either way.

## The four demos

The four scenarios are what `make demo` walks through against the cloud (and
`make demo-local` walks through offline). The granular `make demoN` and
`make demo3-stepN` targets shown below drive each pillar by hand on the local
path, which is handy when you want to linger on a single rule.

### Demo 1: The Tax

**The claim:** with no filtering, you pay for every byte at the door, before
anyone knows if it is worth anything. This is the control group, and for most
teams it is also the current architecture.

```bash
make demo1
make sims
```

**Watch:** the hot lane tracks the raw lane byte for byte. Reduction sits at
0%. The dollar meter climbs at the full ingest rate and never slows down.

### Demo 2: The Audit

**The claim:** you can prove what fraction of your ingest is garbage in about
30 minutes, using nothing but volume-versus-queries data. The pipeline parses
all four formats at the edge and tees a classified sample to disk;
`make audit` joins volume against query-likelihood rules.

```bash
make demo2
make sims
make audit
```

**Watch:** `make audit` prints the volume-vs-queried table and the garbage
ratio: which patterns are billed constantly and queried never. The rules are
heuristics standing in for your real SIEM query logs, documented honestly in
`scripts/audit.py` and `scripts/classify.py`. In production you would run the
same join against your vendor's usage data.

### Demo 3: The Filter

**The claim:** filtering at the source cuts 30%+ of volume without touching a
single log anyone queries. This is THE demo of the series, applied one rule
at a time so you can see what each filter is worth. Every rule operates on
parsed real formats: health-check paths in NCSA and ingress lines, klog and
zap severities in CRI payloads, duplicate crash-loop signatures.

```bash
make demo3-step1   # drop health checks (NCSA paths, k8s probes)
make demo3-step2   # + drop debug-level chatter
make demo3-step3   # + dedupe crash-loop duplicates (same zap caller+msg)
make demo3-step4   # + keep 100% of error/warn/slow, sample 10% of the rest
make demo3         # shortcut for step 4
make sims          # if the streams are not already running
```

**Watch:** each step bends the hot lane down while the raw lane keeps
climbing. At step 4 the reduction clears 30% and every error and warning
line in the source streams is still in the hot lane. You lost volume, not
signal.

### Demo 4: The Tiers

**The claim:** compliance never said keep it in hot storage. Retention-only
logs (CloudTrail events, auth-service logs, login attempts) belong in object
storage at about $0.023/GB-month, not in your SIEM at ingest prices, and you
can still pull a window back out the day the auditor calls.

```bash
make demo4
make sims
make rehydrate FROM=2026-06-10T19:00:00Z TO=2026-06-10T20:00:00Z
```

**Watch:** lines route three ways. Junk is dropped, signal goes hot,
compliance-only lines go cold as gzip files partitioned by hour under
`cold-storage/`. The cold meter reads pennies per month next to the hot
meter. The rehydrate command pulls the requested window back out, raw lines
verbatim (cold storage keeps original bytes, so the window is selected by
receive-hour partition; add `GREP='POST /login'` to narrow).

## Pricing presets

The dollar meter uses `presets.json`. These are **editable estimates, list
prices as of June 2026**, not quotes:

| Key | Default | Basis |
|---|---|---|
| `hot_per_gb_ingest` | 0.10 | Datadog-style $/GB ingest |
| `hot_per_million_events` | 1.70 | per million events, 15-day indexing |
| `cold_per_gb_month` | 0.023 | S3 standard, $/GB-month |

Your vendor, your discount, your numbers: edit the file and restart the board
(`PRESETS_FILE=my-presets.json make board`, or `--presets path`). If you think
a default is wrong, the file is one PR away.

## Run this on a real fleet

The cloud quickstart above already is the real-fleet motion. The only thing
that changes at fleet size is how many nodes you bootstrap: run the Expanso Edge
installer where the telemetry is generated, and each node registers with the
same network. Two details carry the demo's logic to production unchanged:

1. **Target with labels.** This demo labels its node `demo=telemetry-cost` and
   the jobs select on it. On a real fleet you label nodes by region,
   environment, or role and widen the selector so the scheduler places the job
   where you want:

   ```yaml
   selector:
     match_labels:
       env: production
       region: us-west
   ```

2. **Watch it run.** `expanso-cli job describe <name> --profile <name>` and the
   Monitoring view in Expanso Cloud show throughput, bytes/sec, and errors per
   node.

That is the whole point of the demo: the filtering logic you just watched cut
the meter is the same artifact you ship to 400 real nodes.
