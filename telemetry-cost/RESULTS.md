# RESULTS.md

Measured numbers from the eval harness, plus how to reproduce them and how to
submit your own.

## How the numbers are produced

Every published number comes from `evals/run_evals.py` (Python stdlib only)
running the demo scenarios end to end against the committed, seeded
[log-simulators](https://github.com/expanso-io/log-simulators) fixtures:

1. Starts a fresh costboard and a fresh local Expanso Edge node
   (`--data-dir .edge-data-eval`).
2. For each scenario, resets the costboard, deploys a file-input variant of
   the demo job that reads the fixture files instead of the live TCP streams,
   and waits until the costboard stats are stable for 3 seconds.
3. Records the stats and checks them against the scenario's acceptance
   criteria, comparing the pipeline's verdicts against
   `scripts/classify.py`, the reference implementation of the same
   classification rules.
4. Writes the full table to `evals/REPORT.md`.

The fixtures are deterministic: four real-format streams generated by
`make fixtures` (`scripts/make_fixtures.sh`), each seeded and time-anchored
(`--seed 42 --backfill 2h --start-time 2026-06-10T16:00:00+00:00`), so the
files are byte-identical on every machine:

| File | Generator | Lines | Format |
|---|---|---|---|
| `fixtures/app.ndjson` | `logsim-app` | 6000 | structured JSON app logs |
| `fixtures/k8s.log` | `logsim-k8s --scenario crash-loop` | 5000 | Kubernetes CRI lines |
| `fixtures/web.log` | `logsim-web` | 6000 | NCSA combined access logs |
| `fixtures/cloudtrail.ndjson` | `logsim-cloud` | 1200 | AWS CloudTrail JSON |

Ground truth is not synthetic tags on the events; it is the reference
classifier run over the raw lines. Because the input is committed and seeded,
anyone running `make eval` on the same fixtures should land on the same
numbers.

What each scenario must prove:

- **S1 Tax:** hot bytes equal raw bytes; reduction is 0%.
- **S2 Audit:** the audit script's garbage ratio lands within 3 points of the
  `classify.py` reference run over the fixtures.
- **S3 Filter (step 4):** reduction of at least 30%, with every
  reference-classified error and warning line present in the hot lane
  (crash-loop dedupe is measured against unique duplicate signatures, not raw
  duplicate counts) and all slow app requests retained.
- **S4 Tiers:** zero retention-only lines leak into the hot sink;
  cold + hot + dropped equals total; rehydrating a window returns exactly the
  contents of the cold partitions in that window, raw lines verbatim.

## PR your own numbers

The default presets are list-price estimates. Your contract is not list price,
and your telemetry mix is not our fixture. Both are inputs, not assumptions:

1. Copy `presets.json`, put in your real (or anonymized) rates.
2. Optionally regenerate the fixtures with your own mix: edit the stream
   counts, rates, or scenarios in `scripts/make_fixtures.sh` (any
   [log-simulators](https://github.com/expanso-io/log-simulators) tool and
   scenario works), or point the demos at your own log samples.
3. Run `make eval`.
4. Open a PR adding a row to the table below with your `evals/REPORT.md`
   summary, the presets you used, and the fixture seed/mix. Anonymous
   "large fintech, ~N nodes" style attribution is fine.

| Who | Telemetry mix | Presets | Filter reduction | Tiers hot reduction | Notes |
|---|---|---|---|---|---|
| _your row here_ | | | | | |

The interesting PRs are the ones where the numbers come out worse than ours.
If your mix breaks a claim from the series, that is exactly what this harness
exists to surface.
