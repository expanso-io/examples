# Contributing

Thanks for looking. This example lives in the
[expanso-io/examples](https://github.com/expanso-io/examples) monorepo under
`telemetry-cost/`; run everything below from that directory. PRs welcome,
especially numbers (see [RESULTS.md](RESULTS.md)), new filter patterns, and
classification-rule realism.

## Demo targets

The front door is cloud-first. `just doctor` checks the toolchain and cloud
connectivity, `just cloud-setup` runs the one-time Expanso Cloud onboarding,
and `just demo` drives the scenarios through your saved demo profile
(`just demo --scenario tax|audit|filter|tiers`, add `--no-pause` for B-roll).
`just demo-local` is the offline path (local `--local` edge node), and is what
CI and the eval harness use. See [QUICKSTART.md](QUICKSTART.md) and
[DEMO.md](DEMO.md). The granular local recipes (`just demo1..demo4`,
`just demo3-step1..4`, `just edge`, `just board`, `just sims`) are still here
for driving a single pillar by hand.

Scripts that back these targets live in `scripts/`: `preflight.sh` (doctor),
`cloud_setup.sh`, and `demo.sh` (the runner). Any change to them must keep the
SAFETY rule: scripts operate ONLY on the profile recorded in `.demo-cloud.env`
or passed as `--profile`, and must never deploy to or mutate a profile the user
already had.

## Running tests and evals

```bash
# Unit and integration tests (pytest is the only non-stdlib dependency)
just test

# Regenerate the committed fixtures (seeded log-simulators output; needs uv)
just fixtures

# Full deterministic eval run; regenerates evals/REPORT.md
just eval
```

`just eval` starts a local Expanso Edge node and the costboard, runs every
scenario against the committed fixtures, and tears everything down. If a run
dies partway, `just clean` removes stray processes and state (including any
running log-simulator streams).

Every job YAML must pass offline validation before it ships:

```bash
expanso-cli job validate jobs/<file>.yaml --offline
```

CI (`.github/workflows/telemetry-cost-ci.yml` at the monorepo root) runs
fixture regeneration, validation, pytest, and the evals on every PR touching
`telemetry-cost/**`. Green CI is required.

## Config style rules

- Every pipeline is a complete Expanso job YAML: `name`, `description`,
  `type: pipeline`, `config` with `input` / `pipeline` / `output`. No bare
  pipeline fragments.
- Base jobs in `jobs/` stay selector-free so they stay portable to the local
  path. The cloud variants in `jobs/cloud/` are generated by `just cloud-jobs`,
  which injects `selector.match_labels.demo: telemetry-cost` at the spec top
  level (a sibling of `name` / `type` / `config`, NOT inside `config`). Both
  the base and the generated variants must pass `job validate --offline`.
- Comments explain WHY a stage exists, not what the syntax does. The comments
  double as the demo talk track; write them like you would say them.
- Enriching Bloblang mappings start with `root = this`. A mapping that only
  assigns `root.x` replaces the whole document, and that bug is silent.
- Classification rule changes go to BOTH places: the shared Bloblang block in
  the job YAMLs and the reference implementation in `scripts/classify.py`.
  The evals compare them; a one-sided change fails.
- Ports are fixed: 8090 (costboard), 8081 (scenario ingest), 5601 (intake
  socket_server), 19010 (edge API). Do not introduce new ones.
- Python is stdlib only (pytest allowed in `tests/`), 3.10+ compatible, and
  ruff-clean.
- Fixtures are generated, never hand-edited: change
  `scripts/make_fixtures.sh` and run `just fixtures`. Seeded and
  time-anchored, so the committed files must be byte-identical on every
  machine.

## Pricing and vendor conduct

- Preset changes need a source: a public pricing page and the date you read
  it. Presets stay labeled as editable estimates.
- No vendor flames. The hot-sink vendors charge for ingest because that is
  the published model; the repo demonstrates the architecture's cost
  behavior, not anyone's villainy. Keep PRs, comments, and copy in that
  spirit.
